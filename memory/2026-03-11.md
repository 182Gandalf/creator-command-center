### Hook Buttons Mobile Fix (ROUND 2 — Root Cause Found)
**Problem:** First hook card buttons still unresponsive on mobile after initial fix.
**Root Cause:** `hooks-manual-input` (with its `generate-btn` having `position:relative; z-index:2`) stayed fully visible after hook generation. The z-index:2 generate button painted on top of hook-card-0 in the stacking context, blocking touch events.
**Fix:**
- After `generateHooks()`, hide `hooks-manual-input` and replace with compact topic chip ("Topic: X  ✏️ Change")
- `initHooksSection()`: also collapses manual input when restoring cached hooks
- `showHooksManualInput()`: re-expands when user clicks Change
- Added `-webkit-tap-highlight-color: transparent` to `.hook-actions button`
- **Lesson:** `position:relative; z-index:2` on a sibling element ABOVE another in flow can block touch events on the lower element if stacking contexts overlap

### View Script Button Fix
**Problem:** "View Script" button did nothing; clicking the date (card) worked.
**Root Cause:** `addEventListener` on dynamically-created innerHTML elements + locally-scoped `_calNavigateEntry` unreliable. Card-level click handler made entire card a nav link (unintended).
**Fix:**
- `calGoToContent(ideaIdx)` — global function, inline `onclick` on button
- Removed card-level click handler entirely
- Date no longer triggers navigation — only explicit button does
- **Lesson:** Always use global functions + inline onclick for dynamic button elements; addEventListener after innerHTML can be unreliable
